# 22. What happens if one Consumer crashes?

# 23. What happens if one Broker crashes?

### Answer

> **If the Leader Broker crashes, Kafka promotes one of its replicas to become the new Leader. This ensures high availability and prevents data loss.**

---

# 24. What is a Leader Partition?

### Answer

> **The Leader Partition is responsible for handling all read and write requests. Producers send messages to the Leader, and consumers read messages from the Leader.**

---

# 25. What is a Replica (Follower) Partition?

### Answer

> **A Replica (Follower) Partition is an exact copy of the Leader Partition. It continuously copies data from the Leader and acts as a backup.**

---

# 26. Why do we need Replication?

### Answer

> **Replication provides fault tolerance, high availability, and protects against data loss. If the Leader Broker fails, a Replica becomes the new Leader.**

---

# 27. What is Replication Factor?

### Answer

> **Replication Factor is the total number of copies of a partition.**

Example:

```text
Replication Factor = 2

Leader  → Broker 1
Replica → Broker 2
```

Meaning:

```text
Total Copies = 2

1 Leader
1 Replica
```

Formula:

```text
Replicas = Replication Factor - 1
```

# 28. Can the Producer write to a Replica?

# 29. Can the Consumer read from a Replica?

### Answer

> **Normally, no. Consumers read from the Leader Partition.**

Yes. I think you're referring to **DLQ (Dead Letter Queue)**, which is a very common Kafka interview question.

### Q: What is DLQ (Dead Letter Queue)?

### Answer

> **A Dead Letter Queue (DLQ) is a special topic where Kafka stores messages that cannot be processed successfully by the consumer.**

### Simple Example

Suppose the producer sends three messages:

```text
Order1
Order2
Order3
```

The consumer processes them:

```text
Order1 ✅ Success

Order2 ❌ Failed (Invalid data)

Order3 ✅ Success
```

Instead of losing **Order2**, it is sent to the **DLQ**.

```text
Producer
    │
    ▼
Orders Topic
    │
    ▼
Consumer
    │
    ├── Order1 ✅ Processed
    ├── Order3 ✅ Processed
    │
    └── Order2 ❌
             │
             ▼
      Dead Letter Queue (DLQ)
```

### Why do we use DLQ?

* ✅ Prevent losing failed messages.
* ✅ Analyze and fix failed messages later.
* ✅ Retry processing after correcting the issue.

### Interview Answer

> **"A Dead Letter Queue (DLQ) is a special Kafka topic where messages that fail processing are stored. This prevents data loss and allows us to analyze or reprocess the failed messages later."**

### Note

Kafka **does not have a built-in DLQ feature**. In Kafka, a DLQ is **typically implemented as another Kafka topic** (for example, `orders-dlq`) where the application sends failed messages. Frameworks like **Spring Kafka** provide support for this pattern, making it easier to implement.


Q : Is DLQ is kind of exception handling ? 
Ans : 
**Yes! Exactly.** ✅

You can think of **DLQ as exception handling for messages.**

When a consumer processes a message:

* ✅ If processing succeeds → Message is processed normally.
* ❌ If processing fails (exception occurs) → Instead of losing the message, it is sent to the **DLQ**.

### Example

Suppose the consumer receives:

```text id="hsmf2d"
Order1
Order2
Order3
```

Processing:

```text id="s31hvi"
Order1  → Success ✅

Order2  → Exception ❌

Order3  → Success ✅
```

Instead of discarding `Order2`:

```text id="s5j04r"
Order2
   │
   ▼
Dead Letter Topic (DLQ)
```

---

### Java analogy

Think of it like this:

```java
try {
    processMessage(message);
} catch (Exception e) {
    sendToDLQ(message);
}
```

So yes, **DLQ is similar to exception handling**, but instead of just logging the exception, you also **save the failed message** so it can be investigated or retried later.

---

### Interview Answer

> **"Yes, you can think of a DLQ as exception handling for messages. If the consumer cannot process a message because of an exception, instead of losing it, the application sends that message to a Dead Letter Queue (or Dead Letter Topic in Kafka) so it can be analyzed or reprocessed later."**

This is a very good way to explain it in an interview because it's simple and technically correct.
